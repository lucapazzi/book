Part III — Configuration Semantics Giving Meaning to Controller States

Part II established the structure of a PWS system: an Assembly of controlled components and a controller that observes and issues commands. But one essential question remained open: what does a controller state actually mean? Classical finite-state machines leave this question to the designer's imagination. PWS makes it explicit. Instead of relying on informal names, each controller state is associated with a precise set of Assembly configurations. This part introduces the formal semantics that enables the Correct-by-Construction methodology, and explores how autonomous behaviour constrains this semantics.

1. Why controller states need a semantics

So far we have described the controller as an ordinary finite-state machine. States, transitions, guards, and actions define its behaviour. However, an important question remains unanswered:

What does a controller state actually mean?

In classical finite-state machines, the meaning of a state is usually left to the designer. A state named Working or Ready suggests an interpretation, but this interpretation exists only informally. Different designers may associate slightly different meanings with the same name. This ambiguity is harmless in systems designed for informal reasoning, but it becomes costly when formal verification is required.

PWS adopts a different approach. Instead of assigning only a textual interpretation, each controller state is associated with a precise set of Assembly configurations. Consequently, the semantics of a controller state becomes explicit and mathematically verifiable.

The shift is profound: semantics moves from the designer's notes into the formal model itself. What was once implicit becomes explicit, and what was once unverifiable becomes subject to systematic analysis.

2. Specified semantics

The first step in designing a PWS controller consists in declaring the intended meaning of each controller state. This meaning is expressed as a constraint over the states of the controlled components, using the same dot notation we introduced for guards: component.state means "the state of component is state".

Example: Cover–Laser Assembly

Consider a simple Cover–Laser assembly. The controller state Working might be specified by the constraint:

laser.On ∧ cover.Closed

This constraint expresses the designer's intention: whenever the controller is in state Working, the Assembly must be in a configuration where the laser is on and the cover is closed. (Recall that laser.On is shorthand for "laser equals On", and similarly cover.Closed means "cover equals Closed".) Notice that this logical AND corresponds to a single Assembly configuration—specifically, the pair (Closed, On) as we enumerated in Section 7 of Part II. The constraint is a formal statement of which Assembly configurations are admissible while the controller occupies this state. It specifies what Assembly conditions the controller state is supposed to represent and enforce. For this reason, we call it the specified semantics.

The specified semantics therefore acts as a behavioural contract associated with the controller state. It makes the designer's expectations explicit before implementation begins.

Design Philosophy

This approach reverses the traditional workflow of finite-state machine design:

  • Traditional approach: design first, verify afterwards.
  • PWS approach: specify the semantics first, then design the controller to match it.

During controller design, every transition must preserve this contract. The specified semantics therefore guides the design itself, rather than being imposed after the controller is complete. This is the foundation of the Correct-by-Construction methodology.

3. Obtained semantics

Declaring the intended semantics is only the first step. The controller, together with the Assembly, actually generates a possibly larger set of reachable configurations. These configurations constitute the obtained semantics.

Unlike the specified semantics, which is chosen by the designer, the obtained semantics is computed from the controller structure. It represents the actual behaviour of the system: the configurations the controller can actually reach, given the Assembly's rules and the controller's transitions and guards.

Verification consists in comparing these two sets:

Ideal situation: obtained semantics = specified semantics.

Two kinds of mismatch can arise. Whenever the obtained semantics contains additional configurations that were not part of the specification, the controller admits behaviours that were not intended; such situations reveal design flaws. Conversely, if the obtained semantics is a strict subset of the specified semantics, the controller is overly restrictive—it never reaches some intended configurations.

The goal of Correct-by-Construction design is to progressively refine the controller until these two sets coincide.

4. Computing configurations

How is the obtained semantics computed? PWS adopts an inductive construction that mirrors the controller structure. Before giving the procedure, we fix the object it manipulates.

An Assembly configuration can be viewed as a possible world: a valuation with as many propositions as there are machines in the Assembly, each proposition fixed to the state actually occupied by its machine.

A controller state s is associated with a set of Assembly configurations—the configurations the Assembly may assume whenever the controller is in s. These are the possible configurations of s; during execution the system occupies exactly one of them at a time. Correct-by-Construction (CbC) design identifies all and only the possible configurations of every state, through the inductive procedure below. At its base lie the initial configurations of the Assembly, computed from the fact that each machine starts out in one—or possibly several—initial states.

Computational Model

For each controller state s, we compute Conf(s) inductively:

1. Base Step: collect all possible initial configurations into Conf(Init).

2. Inductive Step: for each transition s → t in the controller, every configuration in Conf(s) is transformed (subject to guards and component capabilities), and the resulting configurations are accumulated into Conf(t).

3. Final result: after all transitions have been processed, the final Conf(s) for each state s represents the obtained semantics—the actual configurations the system can reach.

Notation. Sem(s) denotes the specified semantics (declared by the designer); Conf(s) denotes the obtained semantics (computed from the controller structure). Verification checks that Conf(s) ⊆ Sem(s)—that the system never leaves the configurations the designer admits.

The set of configurations associated with each controller state is transmitted to other states through the transitions of the controller. If the controller moves from s to t without directing any action toward the Assembly, t inherits the same set of configurations as s—together with whatever is contributed by every other state entering t. If the transition does carry actions, some configurations are transformed into other configurations; these may already belong to the target state's set, or may not, in which case they are added to it.

Base Step

The initial controller state is associated with the initial Assembly configuration, formed by combining the initial states of all controlled components. For the Cover–Laser example, suppose the controller begins in state Init. Unlike a laboratory setting where we can guarantee exact initial conditions, a real system may not know the initial state of each component: the cover might be either Open or Closed when the system starts (we do not control which), and likewise the laser might be either Off or On.

This uncertainty creates multiple possible initial configurations. Rather than assuming a single initial state, the Base Step must account for all possible combinations of initial component states. Formally, the initial conditions can be expressed as:

(cover.Open ∨ cover.Closed) ∧ (laser.Off ∨ laser.On)

Expanding this expression to canonical form (Disjunctive Normal Form), we obtain four disjuncts:

(cover.Open ∧ laser.Off) ∨ (cover.Open ∧ laser.On) ∨ (cover.Closed ∧ laser.Off) ∨ (cover.Closed ∧ laser.On)

Each disjunct corresponds to one initial configuration. This is the canonical form derived from the product of possible component states.

For the Cover–Laser assembly, there are therefore four possible initial configurations:

  1. (Open, Off) — corresponds to (cover.Open ∧ laser.Off)
  2. (Open, On) — corresponds to (cover.Open ∧ laser.On)
  3. (Closed, Off) — corresponds to (cover.Closed ∧ laser.Off)
  4. (Closed, On) — corresponds to (cover.Closed ∧ laser.On)

All four configurations are added to Conf(Init), establishing the starting point of the inductive construction. This multiplication of configurations—from two possible states of the cover times two possible states of the laser—illustrates why real systems must account for all possible initial conditions, not just the "happy path" scenario.

Inductive Step

Suppose the semantics of controller state X has already been computed. Consider one outgoing controller transition:

X → Y

This transition is associated with actions—commands sent to the controlled components (e.g., <laser.on>, <cover.unlock>). For each configuration c in Conf(X), the transformation proceeds as follows:

  1. Apply the actions associated with the transition to the components in configuration c.
  2. Each action induces a state change in the targeted component (e.g., issuing <laser.on> transforms a laser in state Off into state On).
  3. The result is a new configuration c' reflecting the component state changes.
  4. Add c' to Conf(Y).

Not all actions succeed on all components. If a component's interface does not include the requested command, the action fails and the configuration remains unchanged. This reflects a real-world principle: the controller proposes; the component disposes.

Configuration Accumulation

A critical aspect of the inductive construction is that configurations are accumulated, not replaced. When step 4 of the Inductive Step adds a new configuration c' to Conf(Y), it does not overwrite the existing configurations in that set. Instead, the new configuration is combined with any configurations already present. Formally:

Conf(Y) := Conf(Y) ∪ {c'}

This distinction matters because a controller state Y might receive configurations from several different predecessor states X₁, X₂, X₃, … through different incoming transitions. Each transition contributes its own transformed configurations, and all of them are merged into Conf(Y).

Example: accumulation from two predecessors. Suppose the controller has states Idle and Working. During the first iteration, a transition Idle → Working issues the action <laser.on>. This transforms configurations (Closed, Off) and (Open, Off) into (Closed, On) and (Open, On), which are added to Conf(Working). Later, another transition (perhaps from a Standby state) reaches Working and contributes the configuration (Closed, Off). This configuration is accumulated, so:

Conf(Working) = {(Closed, On), (Open, On), (Closed, Off)}

The inductive procedure repeats over every transition in the controller until no new configurations are added to any state. This iteration reaches a fixed point: a state where Conf(s) no longer changes for any state s. At this point, the inductive construction terminates, and the final Conf(s) for each state represents the obtained semantics—the complete set of reachable configurations.

The existence and computability of this fixed point rests on a fundamental property: the operation of adding configurations to states is monotone. In each iteration we only add configurations to Conf(s); we never remove them. This means the sequence Conf₀(s) ⊆ Conf₁(s) ⊆ Conf₂(s) ⊆ … forms a monotonically increasing chain of sets. By Tarski's fixed-point theorem, any monotone function over a partially ordered set (here, the lattice of configuration sets) must have a least fixed point. Consequently, the inductive procedure is guaranteed to terminate and reach the minimal fixed point—the smallest set of configurations consistent with all controller transitions.

This inductive process mirrors how the controller actually evolves: by examining each possible transition from each reachable state, we discover all configurations the controller can reach.

Configurations as Possible Worlds

The construction above admits an equivalent reading in terms of possible worlds, a notion familiar from modal logic and already anticipated at the start of this section. Recall that a configuration is a valuation assigning to each machine of the Assembly the state it currently occupies; under this reading, a configuration simply is a possible world, and Conf(s) is the set of worlds accessible from controller state s—the worlds the Assembly may inhabit whenever the controller sits in s.

The Base Step populates Conf(Init) with every world consistent with the (possibly uncertain) initial state of each machine—exactly the worlds satisfying the initial formula, expressed earlier in Disjunctive Normal Form as a disjunction of fully specified worlds.

Each controller transition s → t then plays the role of an accessibility relation between worlds. When the transition carries no action, it behaves as the identity relation: every world accessible from s is also accessible from t, so Conf(t) inherits Conf(s) unchanged (merged, as always, with whatever other predecessor states contribute). When the transition carries actions, it relates each world c in Conf(s) to the world c' obtained by updating the valuation of the targeted machines; c' is added to Conf(t) whether or not it was already present, since configurations accumulate rather than replace one another.

Read this way, the inductive construction is the standard modal-logic image operation: Conf(t) is the union, over every incoming transition, of the image of the source state's configurations under that transition's accessibility relation. The fixed point reached at the end of the procedure is the smallest set of worlds closed under all these relations—precisely the worlds the controller can force the Assembly into, state by state.

5. Autonomous behaviour

So far, we have considered only controller-triggered behaviour. Real systems, however, evolve autonomously. Sensors change state. Doors move. Temperatures increase. Timeouts expire. These autonomous transitions modify the Assembly even when the controller remains in the same state.

Consequently, the semantics of controller states cannot ignore autonomous evolution. The presence of autonomous transitions introduces a new constraint on the semantics:

The semantics of a controller state must remain valid even as the Assembly evolves autonomously.

This simple observation is central to understanding why PWS requires the concepts introduced in the next section.

Autonomous Evolution as a Move Between Possible Worlds

In the possible-worlds reading introduced earlier, an autonomous evolution is a move from one possible world to another: the world c the Assembly actually occupies—an element of Conf(s)—is transformed by an autonomous transition into a different world c'. Nothing guarantees that c' is among the worlds the specification expects: Sem(s) only lists the worlds the designer admits and looks for in state s, and autonomous evolution is free to carry the Assembly into a world c' that falls outside that set.

When c' is not in Sem(s), the controller state and the actual world are no longer consistent—precisely the situation the Cover–Laser example in the next section will make concrete. This inconsistency is what triggers a controller transition in response. Crucially, that reaction need not simply accept c' as it stands: like any controller transition, it may carry actions of its own, and so it can transform c' further, into a world c''—possibly different from c'—which is the world the Assembly ends up in once the controller has reacted.

Read this way, autonomous behaviour and controller reaction compose two successive moves between possible worlds: an uncontrolled move c → c', driven by the Assembly itself, followed by a controlled move c' → c'', driven by the controller's reactive transition. Section 6 studies exactly how this second move is computed, and what it takes for the controller to guarantee that such a reaction always exists.

6. The reactive space

Suppose the controller is currently in state Working, whose specified semantics requires:

laser.On ∧ cover.Closed

Equivalently, in tuple notation: (Closed, On). Now suppose that the cover autonomously changes from Closed to Open. Immediately after this autonomous transition, the controller is still in state Working, while the Assembly configuration has become:

laser.On ∧ cover.Open

Equivalently, in tuple notation: (Open, On).

The controller state and the Assembly configuration are no longer consistent. The Assembly has moved outside the semantics of the controller state.

This observation motivates one of the central ideas of the PWS methodology:

Whenever autonomous behaviour moves the Assembly outside the semantics of the current controller state, the controller must react by moving to another state whose semantics is compatible with the new configuration.

The set of configurations to which the Assembly can move autonomously from the current controller state—and for which a suitable controller reaction exists—is called the Reactive Space. Understanding reactive spaces, and ensuring they are properly handled, is the key to building controllers that remain correct despite autonomous evolution.

What Lies Ahead

The formal treatment of reactive spaces, and the mechanisms the controller must employ to handle them correctly, constitute the foundation of the Correct-by-Construction methodology. This is the subject of Part IV.

For now, it suffices to note that by introducing explicit semantics we have identified a new and unavoidable constraint on controller design: the controller must not only reach its intended configurations, but must remain responsive to autonomous behaviour that might move the Assembly beyond its current state's semantics.

This is the first time in this book that we have encountered a design problem that cannot be solved by applying the guard-and-transition logic of a classical state machine. It is this problem that Correct-by-Construction design, and the PWSEditor tool, are built to solve.
