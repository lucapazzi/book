Part IV — Designing Correct Controllers A Correct-by-Construction Methodology

Part III showed how each controller state carries explicit semantics: a set of Assembly configurations it is supposed to represent. But how does the designer actually build a controller that matches this semantics? Part II and III posed the problem; this part provides the solution. Rather than designing first and verifying afterwards, the PWS methodology inverts the process: semantics guides design from the start. The controller emerges as the mechanism that realizes intended semantic properties, not as a finished artifact to be validated. This is the Correct-by-Construction philosophy.

1. From behaviour to semantics

A controller is not designed by drawing states. It is designed by progressively realizing their intended semantics.

Traditional state-machine design usually follows a well-established pattern:

  1. Draw states and transitions.
  2. Complete the behaviour description.
  3. Afterwards, ask whether the resulting system satisfies the intended requirements.

This workflow separates design from verification: behaviour comes first, correctness comes later.

PWS follows the opposite philosophy. The meaning of every controller state is declared before the controller itself is completed. The semantics therefore becomes the primary design artifact. The controller is merely the mechanism used to realize that semantics.

This inversion of perspective is the key idea underlying Correct-by-Construction. Design and verification are no longer separate phases. They become integrated into a single iterative process where semantic constraints guide every design decision.

2. Declaring the intended behaviour

The design process begins with a crucial step: associating every controller state with its specified semantics.

These semantic constraints describe which Assembly configurations are considered compatible with each controller state. For a Cover–Laser assembly, for example:

  • The Idle state might specify (laser.Off, cover.Closed) — where laser.Off means "laser equals Off" and cover.Closed means "cover equals Closed"
  • The Ready state might specify (laser.Off, cover.Open)
  • The Working state might specify (laser.On, cover.Closed)

At this stage, the controller may still contain very few transitions. The objective is not yet to describe the behaviour in detail. Instead, it is to describe the intended meaning of each state. The semantics acts as a behavioural contract: a promise that certain configurations will be reached and others will be avoided.

This declaration marks a profound shift in design methodology. The designer is forced to be explicit about what each state means before implementing the transitions that lead to it.

3. Constructing behaviour

Once the intended semantics has been declared, controller transitions are progressively introduced.

Each transition contributes new Assembly configurations to its destination state. Consequently, controller transitions are not merely graphical connections. Each transition represents a semantic transformation: it expands the set of configurations reachable in the destination state.

After every design step, the obtained semantics can be recomputed. The designer immediately observes the semantic consequences of the newly introduced transition. Design therefore becomes an iterative dialogue between the controller structure and its semantics.

Instead of completing a design and then discovering violations of requirements, the designer learns the consequences of each choice as soon as it is made. Semantic inconsistencies are detected early, when modifications are still easy to introduce.

4. The design cycle

The complete Correct-by-Construction methodology can be summarized by a simple iterative process:

  1. Specify the intended semantics for each controller state.
  2. Construct controller transitions that drive the Assembly.
  3. Compute the obtained semantics from the controller structure.
  4. Compare obtained and specified semantics.
  5. Identify semantic inconsistencies and unintended behaviours.
  6. Introduce repair transitions to handle unexpected evolution.
  7. Verify the local obligations governing reactive behaviour.
  8. Repeat until both semantics coincide.

Correctness is therefore not verified after design. It progressively emerges during design. This design philosophy constitutes the central contribution of the PWS methodology.

5. Autonomous evolution

Real systems continue to evolve even when the controller remains in the same state. Sensors change value. Mechanical devices move. Timeouts expire. Environmental conditions vary.

These autonomous transitions may invalidate the semantics associated with the current controller state. We have already encountered this problem in Part III: the Cover might autonomously change from Closed to Open while the controller is still in a state that expects cover.Closed.

Autonomous evolution creates a new design requirement. The controller must accomplish two different tasks:

  • Drive: Actively move the Assembly towards its intended behaviour.
  • React: Continuously respond to autonomous changes that may compromise semantic consistency.

A controller that only drives but does not react will not remain correct in the presence of autonomous evolution. Conversely, a controller that only reacts but cannot drive will never make progress towards its goals. Both capabilities are essential.

6. Semantic repair

Whenever autonomous evolution produces an Assembly configuration that no longer belongs to the semantics of the current controller state, the controller must react.

Its objective is not simply to execute another transition. Its objective is to restore consistency between

  • the current controller state, and
  • the current Assembly configuration.

Controller transitions triggered by observed state changes therefore become semantic repair mechanisms. The controller continuously maintains the correspondence between its own state and the actual state of the Assembly.

This is a fundamentally different role from the driving transitions discussed earlier. Repair transitions are reactive: they respond to events, not to the designer's intention to progress. They restore an invariant that must hold at all times: every reachable configuration must belong to the semantics of the current controller state.

Design complexity increases significantly when repair transitions must be introduced. The designer can no longer assume that the controller can ignore autonomous changes. Every controller state must account for all possible autonomous evolution.

7. Reactive spaces

Reactive behaviour naturally introduces a new concept: the Reactive Space of a controller state.

For a controller state X, the Reactive Space consists of all Assembly configurations that become reachable through autonomous evolution while the controller still remains in X.

Some of these configurations remain compatible with the specified semantics of X. Others require controller intervention. Reactive Spaces therefore identify precisely where repair transitions become necessary.

Rather than representing failures, Reactive Spaces represent situations in which the controller is expected to react:

A Reactive Space reveals the set of configurations from which the controller must be able to move to another state without losing correctness.

The discovery and analysis of Reactive Spaces is where the Correct-by-Construction methodology becomes most powerful. The methodology tells the designer exactly which autonomous transitions must be handled, and which controller transitions are required to handle them.

8. Well-formed controllers

Not every controller can be analysed using the methodology introduced in this part. Before semantic verification becomes meaningful, the controller must satisfy a collection of structural requirements.

These requirements are expressed by the Well-Formedness Rules. Together, they guarantee that:

  • state semantics are well-defined,
  • obtained semantics are computable,
  • reactive situations are completely handled,
  • semantic ambiguities cannot arise.

The Well-Formedness Rules therefore characterize the class of controllers for which Correct-by-Construction applies.

These rules are not arbitrary constraints imposed from outside. Rather, they are the minimal requirements needed to ensure that the semantic model is coherent and the design cycle can proceed without ambiguity. A controller that violates the rules may still execute correctly, but its semantics cannot be analyzed using the PWS methodology.

Well-formedness ensures that the designer can rely on the semantic analysis performed by PWSEditor. Without these rules, the tool's guarantees would be limited to syntactic checks, and semantic properties could not be verified.

The Correct-by-Construction methodology transforms controller design from an art into a discipline. By making semantics explicit and central, it provides a systematic way to build reactive controllers whose correctness emerges from the design process itself, rather than being established after the fact through exhaustive testing or formal verification.
