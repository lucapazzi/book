Part VI — The Practice Building a Controller with PWSEditor

So far we have done theory: the declared semantics Sem, the accumulated semantics Semacc, the reactive space RS, exit zones, and the Correct-by-Construction (CbC) obligations. In this chapter we put it all to work. We will not present PWSEditor as «a program whose buttons we list», but as the tool that makes visible what we have learned to compute by hand — and that, in doing so, guides the design.

Together, step by step, we will build the Cover + Laser controller. We will do it the way the tool invites us to work: we sketch an intention, the tool shows us the semantic truth by coloring the dashboards, and we close the red until the model is correct. This red → green cycle is the true engine of the method.

1. What PWSEditor does (and doesn’t do)

The problem. We have learned that each controller state corresponds to a contract (Sem) and a set of physically reachable configurations (Semacc), computed as a least fixed point. No designer wants to iterate the Kleene operator by hand every time a transition is added.

The insight. PWSEditor computes these semantics continuously, on every change to the model, and paints the result. We draw the intent (the states, their contracts, the transitions); the tool tells us what actually happens. In this paradigm, designing means making the red disappear.

What it does

For each controller state, PWSEditor computes and shows:

  • the declared domain (Sem, the constraints we impose on the state);
  • the accumulated reachable configurations (Acc);
  • the reactive space (RS) and the exit zones, i.e. the autonomous evolutions of the components that would leave the state’s contract.

It also automatically flags formal violations: deadlocks (primary and secondary), uncovered exit zones, and constraint violations. Every time we touch the model, the computation is re-run.

What it does not do (honesty about the prototype’s state)

  • It doesn’t design for us. The tool lists the outstanding obligations; we choose the right reaction.
  • The report’s LTL verification section is, at present, a placeholder («coming soon»): temporal logic is the direction, not yet an operational feature.
  • Export to Structured Text / PLCopen exists and works, but currently only for simple component machines; nested (compositional) assemblies are not yet exported.

2. Anatomy of the interface

The PWSEditor window is divided into three main areas:

  • Left — Controller Editor. The controller canvas: states (circles), the initial pseudo-state (small filled black circle), transitions (arrows), floating annotations (guards, actions, semantics), and state dashboards.
  • Top right — Assembly / Library. The list of assembly components. At the bottom of the Assembly view is the initial-configurations dashboard, showing the drift-closure of the system at startup.
  • Bottom right — Embedded editor. Opens on double-click on a component; used to edit its states and transitions.

Reading the state dashboard

Each controller state has a floating dashboard with a colored name band (green / gold / red) and three sections:

SectionContent
SemThe user-declared contract (rows in blue)
AccThe accumulated configurations — those actually reachable in the state (green rows = satisfy the constraints, red = violate)
RSThe exit zones, one per row: autonomous evolutions that would leave the contract

The border color summarizes the health of the state:

BorderMeaning
GreenEverything is fine — no issues
Gold / yellowState is unreachable (empty semantics)
RedThere are issues to resolve

Exit zones have their own color code:  covered,  uncovered or orphaned,  internal (no reaction required),  provisional,  coverage not required (fail state).

Graphical notation and creation gestures

3. Step 0 — Modeling the assembly (Cover + Laser)

The problem. Before the controller come the parts: the physical components with their local behavior, often under-specified.

Example. Our plant has two devices:

  • cover — a mechanical cover, observable, with states Open and Closed, which can move autonomously (the environment opens or closes it);
  • laser — a device with states Off and On, which changes state only when the controller emits the triggers on / off (no autonomous behavior).

Realization in PWSEditor.

  1. In the right-hand panel select the Assembly view, then Add → create the machine with ID cover.
  2. Double-click on cover: the embedded editor opens. Use right-click → Add State to create Open and Closed.
  3. Connect the two states with two autonomous transitions (Cmd/Ctrl + drag): Closed → Open and Open → Closed.
  4. Set the cover’s initial state: from the pseudo-state create two initial transitions, one to Open and one to Closed. At startup the position is unconstrained.
  5. Back in Assembly, Add → machine laser, states Off and On.
  6. The laser’s transitions are event-triggered: create Off → On with event on and On → Off with event off (Cmd/Ctrl + Shift + drag, then double-click the label to rename).

Note the initial-configurations dashboard at the bottom of the Assembly view: this is the semantics of the assembly taken as a whole, i.e. the configurations the system can be in right after power-on. In our case: {(cover.Closed, laser.Off), (cover.Open, laser.Off)}. The rows appear in red because the controller is still empty: no initial transition covers them yet.

4. Step 1 — A first reactive controller

The insight. Before making the laser «do work», the controller must be able to keep pace with the environment. So we start from the simplest possible behavior: a controller that merely follows the cover, without turning anything on. Just two states, no safety concerns yet — the ideal setting to meet transitions for the first time.

Two states, with the laser always off:

StateContract (Sem)Meaning
Idle(cover.Open, laser.Off)cover open, laser off
Ready(cover.Closed, laser.Off)cover closed, laser off

The laser stays in the assembly, but for now it is never commanded: both contracts keep it Off. We will turn it on only later (Step 4).

Realization in PWSEditor.

  1. In the left canvas, right-click → Add State: create Idle and Ready.
  2. Declare the contracts (right-click on the dashboard → Edit Constraints Semantics).
  3. Create the initial transitions from the pseudo-state: [(cover.Open)] → Idle and [(cover.Closed)] → Ready.

At this point we stop: no transition between Idle and Ready. The model is still incomplete — and that is exactly what we want the tool to tell us.

5. Step 2 — Reading the colors: exit zones

The problem. Both dashboards are red. This is not an error: the tool is listing the obligations still open.

Where the red comes from. The cover moves autonomously, and each of the two contracts fixes its position. When the cover changes, the system leaves the state’s contract: this is an exit zone.

  • In Idle (cover.Open) the cover can do Open → Closed: the exit zone cover: Open→Closed arises. No transition covers it → red.
  • In Ready (cover.Closed) the cover can do Closed → Open: exit zone cover: Closed→Open, also uncovered → red.

These two exit zones are entirely harmless from a safety standpoint (the laser is off in both cases): they only serve to coordinate the controller with the environment. That is exactly why they are the ideal example for understanding the mechanism, before meeting a critical one.

How to read them in the tool. Each exit zone appears as a row beneath the dashboard’s matrix, labeled cover:Open→Closed. Red flags «uncovered». For the full list: right-click on an empty area of the canvas → Controller Report..., section Uncovered Exit Zones.

6. Step 3 — Covering the exit zones: the first transitions

The insight. For the controller, an autonomous transition of the cover is not a command to issue, but an event to observe. Each exit zone carries the obligation of a reaction: a guard-triggered transition that fires as soon as the observed condition becomes true. Let’s cover them one at a time, so we can see the effect of each.

First transition. Add Idle → Ready with guard [cover.Closed] (Cmd/Ctrl + drag from Idle to Ready; then pick the proposition from the guard menu, which lists exactly the state’s exit zones). As soon as it is created, Idle’s exit zone turns from red to green: the cover closing is now handled. Ready’s exit zone, however, is still red.

Second transition. Add the symmetric Ready → Idle with guard [cover.Open]. The second exit zone also turns green. Both dashboards are now green: the reactive controller is complete with respect to the cover’s drift.

These two transitions are the Idle↔Ready movement that follows the cover: we did not add them «by hand» as an a priori idea, but the tool requested them, exit zone after exit zone. This is the first, simplest use of reactive transitions.

Caution. A newly created guard-triggered transition is born with guard FALSE (a placeholder, shown in red): until you set it, it «never fires». The guard menu proposes the propositions of the available exit zones.

7. Step 4 — The operational leap: Work, the laser, and the fail-safe

So far the controller only observes. Now we make it operate: we add the working mode in which the laser is on — and with it, the first safety constraint.

The new state. Add Work with contract (cover.Closed, laser.On). The safety contract, now explicit, is: the laser must never be on while the cover is open — no state admits (cover.Open, laser.On).

The operational transitions (event-triggered, Cmd/Ctrl + Shift + drag):

  • Ready → Work with event start and action ⟨laser.on⟩ (turns the laser on);
  • Work → Ready with event stop and action ⟨laser.off⟩ (turns it off).

As soon as Work exists, the tool flags a new red — but this time not harmless. In Work the cover can still open autonomously (Closed → Open): the drift would lead to (cover.Open, laser.On), i.e. laser on with the cover open. It is an uncovered exit zone and a violation of the safety contract.

The fail-safe reaction. When the cover opens, the controller cannot close it again (it has no commands over the cover): the only safe reaction is to immediately turn the laser off and return to a safe state. Add Work → Idle (guard-triggered) with guard [cover.Open] and action ⟨laser.off⟩.

Here the strict action semantics comes into play: the emission ⟨laser.off⟩ is transferred only to the configurations in which off is enabled on the laser — that is, exactly those with laser.On, precisely the exit zone’s configuration. The dangerous configuration does not «stay» in Work: it is carried back to safety. The exit zone turns green, and the violation disappears.

8. Step 5 — Closing the model

A controller is complete with respect to its declared contracts when every exit zone is covered, no deadlocks exist, and no computed configuration violates the constraints. This is the operational meaning of CbC closure: many small local checks instead of one large global verification.

With the reactions to the cover’s drift (Idle↔Ready) and the fail-safe of Work, all dashboards turn green and the Controller Report reports CONTROLLER IS WELL-FORMED.

Notice the property we actually cared about: the configuration (cover.Open, laser.On) never appears among the reachable configurations. Safety was not verified after the fact: it is true by construction, because the method never let us leave uncovered the exit zone that would have produced it.

9. Operational checklist and common mistakes

The workflow, in summary

  • Draw the assembly — components, interface states, autonomous (environment) vs. event-triggered (commandable) transitions.
  • Start from the reactive behavior — few states, declared contracts, still no transitions: let the tool show you the exit zones.
  • Cover the exit zones one at a time with guard-triggered transitions, checking red→green at every step.
  • Add the operational layer — event-triggered states and transitions (this is where actions and strict semantics come in), and the fail-safe wherever safety requires it.
  • Recheck the Controller Report until it is well-formed.
  • Export (ST/PLCopen) for simple components.

Common mistakes to spot immediately

  • Fake exit / strict semantics. An event-triggered transition that emits an action not enabled in certain source configurations silently discards them: the arrow is there graphically, but it doesn’t cover the whole spectrum. Symptom: a red underline (secondary deadlock) or a destination state with a gold border (unreachable because the action discarded everything).
  • Red FALSE guard on an autonomous transition: you forgot to set the condition.
  • Orphaned guard (red): the referenced exit zone has changed; update the guard.
  • Overlapping / incomplete event guards (red / orange): they violate determinism or completeness of the partition for that (source, event) pair.
  • Gold border = unreachable state: often caused by strict actions discarding every candidate configuration.
  • Partial configurations. If a constraint doesn’t mention a component, that component is «anything»: its autonomous moves are internal evolution (grey), not exit zones. Constrain only what you actually need.

10. Exercises

1. Cover + Lamp (warm-up). Replace the laser with a lamp that has no safety constraint. Remodel Work as (cover.Closed, lamp.On) but without the prohibition «lamp on with cover open». Observe in the dashboards that the cover’s drift still generates coordination exit zones, but no longer dangerous violations. This isolates what changes when the safety constraint is removed.

2. Fan with timeout. Introduce a fan and turn Work into a timed state (right-click → Timed state, then Edit timed label... with 10s). Add its timeout transition to a Cooldown state. Observe that the timeout, like autonomous moves, contributes to the reactive evolution and hence to the exit-zone computation. Add a fail state (right-click → Fail state, dashed yellow border) and verify that exit-zone coverage is not required for it.

3. A second safety device. Add a component that shares the cover with the laser. Notice how the coverage obligations multiply and how the number of reactions grows: this is the concrete motivation for composition — the topic of the next chapter.
