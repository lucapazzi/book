Part I — Introduction What Makes PWS Different

Reactive systems surround us. Air traffic control systems, medical devices, automotive controllers, industrial robots—these are systems that must continuously observe their environment, react to events, and maintain correctness under all circumstances. For decades, engineers have approached the design of such systems using a standard two-phase workflow: design first, verify afterwards. Part-Whole Statecharts offers a radically different approach. Rather than treating verification as a final validation step, PWS integrates formal reasoning into the design process itself. This introduction explores why this shift matters, and how it changes the way we think about building reactive systems.

The Traditional Approach

Designing reactive systems has traditionally been regarded as a two-stage process.

  1. First, a controller is designed to achieve the desired behaviour.
  2. Only afterwards is the resulting model verified against its functional and temporal requirements using model checking or other formal verification techniques.

This workflow has proved extremely successful. For decades, it has been the dominant paradigm in reactive system design. Thousands of industrial systems have been built using this approach.

However, this workflow also exhibits an inherent limitation:

Verification remains a separate activity performed after the controller has already been designed. Whenever verification fails, the designer must revise the controller and repeat the process, often through several iterations.

This creates a fundamental asymmetry in the design process. Designers invest effort building the controller structure first, and only then learn whether their choices were correct. By that point, significant rework may be necessary.

Traditional Workflow

Design → Build controller

Verify → Check properties

If fail: Redesign and repeat

PWS Workflow

Specify → Declare semantics

Construct → Build incrementally

Verify → Check as you go

A Different Philosophy

The philosophy underlying Part-Whole Statecharts is fundamentally different.

Rather than treating verification as a final validation step, PWS integrates formal reasoning into the design process itself.

Instead of the design-then-verify workflow, PWS follows a semantics-first approach:

  1. The designer specifies the intended semantics of the controller states before building transitions.
  2. The controller is progressively constructed so that this semantics is realized.
  3. Local correctness obligations are verified as the design evolves.

Correctness therefore emerges incrementally rather than being assessed only after the controller has been completed.

More Than a Formalism

For this reason, PWS should not be regarded merely as a formalism for modelling or verifying reactive systems. It is, first and foremost, a design methodology.

PWS comprises four complementary components:

Modelling Language
Provides the vocabulary needed to describe reactive systems hierarchically, with explicit assemblies and controllers.
Configuration Semantics
Gives precise meaning to controller states, making semantics explicit instead of informal.
Correct-by-Construction
Guides incremental controller development so that correctness emerges during design, not after.
Compositional Verification
Establishes global correctness by combining locally verified results according to hierarchical organization.

The Structure of This Book

This tutorial follows the same philosophy as the PWS methodology itself: rather than presenting definitions in isolation, it accompanies you through the complete engineering process.

The Journey Ahead

Part II — The PWS Model introduces the fundamental concepts: assemblies, components, controllers, and the notion of explicit interfaces. You will learn how to think about reactive systems as hierarchies of parts and wholes.

Part III — Configuration Semantics shows why controller states need explicit meaning. You will see how every state is associated with a set of configurations, and how this semantic foundation guides design.

Part IV — Designing Correct Controllers presents the Correct-by-Construction methodology. Rather than drawing controllers and hoping they are correct, you will learn to design them semantics-first, constructing behaviour incrementally while verification emerges.

Part V — Compositional Verification shows how hierarchical systems can be verified hierarchically. Instead of building a global state space that explodes exponentially, verification follows the same hierarchical decomposition as design.

Postface — Reflections steps back to consider what we have learned, and places PWS within the broader context of reactive system design.

Objective

The objective of this tutorial is not simply to explain the PWS formalism. Rather, it is to teach a systematic way of designing reactive controllers whose correctness progressively emerges from the design itself.

If you finish this book thinking differently about how controller states should be designed—if you find yourself asking

"What should this state mean?"

rather than

"Which transition should I draw next?"

then this tutorial will have achieved its principal objective.

The chapters that follow present a single, coherent vision of reactive system design: one in which modelling, semantics, construction, and verification are not separate concerns, but aspects of a unified methodology.
