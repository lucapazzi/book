Part V — Compositional Verification Verifying Hierarchical Systems

Parts II through IV have shown how to design controllers that respect their declared semantics. But how do we verify correctness at scale? Classical model checking generates the complete global state space of the entire system—a process that succumbs to exponential state explosion as systems grow. PWS offers a radically different approach: rather than verifying the global system, verification follows the hierarchical structure of the controller hierarchy itself. Each controller is verified together with its directly controlled Assembly. Once verified, that subsystem becomes an abstract component for the level above. This part explores how compositional verification transforms intractable global problems into manageable local ones.

1. Why verification can be compositional

Traditional model checking considers the complete system as a single transition system. The global state space is generated by composing all the component machines and the controller into one behavioural model. Properties are then verified over this global model.

Although this approach is completely general, it suffers from the well-known state explosion problem. The number of reachable states grows exponentially with the number of interacting components. For a system with n components, each with an average of s states, the global state space contains approximately sn states. As systems grow from dozens of components to hundreds, verification becomes computationally infeasible.

The philosophy adopted by PWS is fundamentally different. Instead of constructing one large behavioural model, the verification process follows the hierarchical organization already present in the system.

Each controller is verified together with the Assembly that it directly coordinates. Once this verification has been completed, the entire subsystem can be regarded as a single verified component by the controller located one level above. Verification therefore follows exactly the same hierarchical organization used during system design.

This alignment between design hierarchy and verification hierarchy is what makes compositional verification possible—and powerful.

2. Contracts

To make hierarchical verification possible, every PWS exports a behavioural contract.

A contract summarizes those aspects of the subsystem that are relevant for its environment, while hiding all internal implementation details. The parent controller therefore interacts with the contract rather than with the internal state machines composing the subsystem.

This abstraction mechanism is essential: without it, the parent would need to understand every detail of every child, negating the benefits of hierarchy. With contracts, the parent only needs to know what the child guarantees to do.

A contract is a formal boundary. On one side lies the subsystem's internal structure; on the other side lies its externally observable behaviour. The contract defines what can be known from outside without revealing what lies within.

3. Contracts in PWS

A PWS contract is associated with the controller representing the whole. The contract does not expose the internal configuration semantics of the controlled Assembly. Instead, it exports only those properties that are observable by the parent controller.

The contract therefore becomes the public behavioural interface of the subsystem.

Internally, the subsystem may contain:

  • many controllers at different hierarchy levels,
  • multiple component machines with complex interactions,
  • elaborate semantic constraints,
  • reactive repair transitions.

Externally, however, it behaves exactly like a single controlled component. This abstraction mechanism allows hierarchical systems to be verified one level at a time, without ever needing to reconstruct the complete global state space.

4. Assumptions and guarantees

Every PWS contract is defined by a pair:

C = (Assumptions, Guarantees)

Assumptions

The assumption part specifies which behaviours are expected from the environment. Assumptions describe the conditions under which the subsystem is expected to operate correctly.

Typical assumptions include:

  • requests eventually terminate,
  • commands are not generated simultaneously,
  • communication channels preserve message ordering,
  • input sensors provide valid readings.

Guarantees

The guarantee part specifies the behaviour provided by the subsystem whenever the assumptions hold. Guarantees describe what the system promises to deliver.

Typical guarantees include:

  • safety properties (something bad never happens),
  • liveness properties (something good eventually happens),
  • sequencing constraints (events occur in the right order),
  • mutual exclusion conditions (conflicting states never coexist).

Notice the fundamental asymmetry: assumptions concern the environment, while guarantees concern the subsystem itself. The contract states: "If the environment behaves this way, we promise to behave that way."

5. Contract composition

Suppose a subsystem L (Lamp) is controlled by another subsystem S (Traffic Light), which in turn belongs to a larger controller C (Intersection Controller).

Lamp (L) ↓ controlled by Traffic Light (S) ↓ controlled by Intersection Controller (C)

Verification proceeds from the leaves of the hierarchy towards the root.

Step 1: Verify L

The Lamp subsystem is verified independently. Its contract exports guarantees about lamp behaviour: for example, a lamp eventually reaches the requested state.

Step 2: Verify S using L's guarantees

The Traffic Light subsystem can now assume L's guarantees. These guarantees become observable facts for S. Using them, S can prove its own higher-level properties: for example, colour sequences follow traffic conventions, and lamp state transitions occur in the expected order.

Step 3: Verify C using S's guarantees

Finally, the Intersection Controller assumes the guarantees exported by the Traffic Lights and proves global safety properties: conflicting traffic directions are never simultaneously enabled.

At no point is it necessary to reconstruct the complete state space of the entire hierarchy. Each level verifies only its own Assembly, making use of the verified properties of the components below.

6. Lifting and hiding

Contracts propagate through the hierarchy by means of two complementary operations:

Lift

Properties proved for a child become assumptions available to its parent. The lift operation translates child properties into the vocabulary of the parent controller.

For example, if the Lamp guarantees "state changes occur within 100 ms," the Traffic Light lifts this guarantee to its own level: "colour sequences respect timing constraints."

Hide

Once a subsystem has been verified, its internal machines become irrelevant to higher hierarchical levels. Only the observable guarantees remain visible. The hide operation removes all internal implementation details while preserving the externally observable behaviour.

For example, the Intersection Controller need not know how the Traffic Light internally manages lamp states. It only needs to know the colour sequence guarantee.

Together, lift and hide transform a verified PWS into an abstract behavioural component suitable for further composition. The internal complexity is encapsulated; the external interface is simplified.

7. Running example

Consider the three-level hierarchy introduced above, with concrete contracts:

Lamp (L) Contract

Assumptions:

  • Request commands are issued sequentially.
  • At most one command per cycle.

Guarantees:

  • Lamp reaches requested state within 50 ms.
  • No spurious state changes.

Traffic Light (S) Contract

Assumptions:

  • Cycle requests are non-overlapping.
  • Lamp guarantees are honoured (lifted from L).

Guarantees:

  • Colour sequence: Red → Green → Yellow → Red.
  • Transitions respect timing (e.g., Green ≥ 10s).

Intersection Controller (C) Contract

Assumptions:

  • Traffic Light guarantees are honoured (lifted from S).

Guarantees:

  • No two perpendicular traffic directions have Green simultaneously.
  • Intersection remains safe under all operating conditions.

Notice how each level assumes the guarantees of the level below, and exports guarantees for the level above. The composition is sound: global correctness follows from local correctness at each level.

8. From local proofs to global correctness

The fundamental principle underlying compositional verification is remarkably simple:

Every controller is responsible only for the correctness of its own Assembly. Whenever this responsibility has been formally discharged, the controller exports a contract summarizing its externally visible behaviour.

Higher hierarchical levels no longer need to know how this behaviour is internally produced. Consequently, the complexity of verification follows the modular organization of the system rather than the combinatorial growth of the global state space.

This is perhaps the most important consequence of the Part–Whole philosophy.

Three Hierarchies Become One

The modelling hierarchy becomes simultaneously:

  • the design hierarchy (how we structure the controller),
  • the semantic hierarchy (how we define correctness at each level),
  • the verification hierarchy (how we prove correctness composionally).

The three perspectives therefore coincide within a single formal framework. This alignment is what makes the PWS methodology so powerful: design, semantics, and verification are not three separate concerns, but aspects of a single coherent process.

What Lies Ahead

Contracts specify what a subsystem guarantees. But how are these guarantees expressed formally? Temporal logic provides the language. Properties such as "a lamp eventually reaches the requested state" or "conflicting directions never have Green simultaneously" are naturally expressed as temporal formulas.

Temporal logic provides the language used to express these guarantees formally. Linear Temporal Logic (LTL) and related formalisms allow complex dynamic properties to be stated precisely, verified mechanically, and linked back to the design hierarchy.

Compositional verification is not merely a pragmatic response to state explosion. It is a fundamental insight: systems designed hierarchically can be verified hierarchically. Design complexity and verification complexity follow the same modular structure. This unity of structure across design, semantics, and verification is what transforms PWS from a notational convenience into a systematic methodology for building and verifying reactive systems.
