Part I — Why The Problem

Before introducing Part–Whole Statecharts, we must understand the problem they were designed to solve. This part builds that motivation from the ground up: what a reactive system is, why finite-state models are the natural language for it, how composition makes the state space explode, what «correct» even means, and why establishing correctness after the fact — the classical verification workflow — is both powerful and, structurally, always late.

1. Introduction

Modern reactive systems are built from many interacting components. Sensors continuously observe the physical environment. Actuators modify it. Communication devices exchange information. Software controllers coordinate the overall behaviour.

Consider a modern elevator, an automated warehouse, or an engine control unit. None of these is difficult because of its individual parts. A motor, a valve, a proximity sensor are, in isolation, well understood. The difficulty lies elsewhere: it lies in coordinating the evolution of many components so that the system as a whole behaves correctly, safely, and efficiently.

It is useful to distinguish two kinds of software. A transformational program reads an input, computes a result, produces an output, and terminates. A reactive system does none of these things in a single step. It never terminates. It maintains an ongoing dialogue with an environment that it does not control and cannot predict.

This has an immediate consequence. Reactive systems are naturally described in terms of behaviour over time, not in terms of a single input-to-output computation. Correctness is a statement about what the system does, moment after moment, in response to whatever the environment happens to do.

2. Behaviour before software

Long before writing a single line of code, engineers reason in terms of situations.

Example: Automatic Door

Consider an automatic door. Its behaviour can be described by four situations: Closed, Opening, Open, and Closing.

These situations change in response to events:

  • a person is detected
  • the door is completely open
  • a timeout has expired
  • the door is completely closed

The relationship between situations and events can be written compactly:

FromEventTo
Closedperson detectedOpening
Openingdoor completely openOpen
Opentimeout expiredClosing
Closingperson detectedOpening
Closingdoor completely closedClosed

Two observations matter here.

First, this description exists independently of any implementation. It can be drawn on a whiteboard and discussed with people who will never read the code. It is a specification, not yet a program.

Second, a situation is an abstraction. «Opening» says nothing about the exact position of the door, the current drawn by the motor, or the temperature of the bearings. A situation is, in effect, an equivalence class of physical states that the system chooses to treat alike. Choosing the right situations is already a design decision.

3. Finite-state machines

Finite-State Machines (FSMs) are among the simplest and most successful abstractions for describing reactive behaviour.

A machine consists of:

  • a set of states,
  • a set of transitions,
  • the events that trigger those transitions.

Optionally, transitions may also produce actions.

FSMs succeed because they are simultaneously intuitive, visual, executable, and analysable. The automatic door above is already a finite-state machine.

Two properties of a machine deserve names. A machine is deterministic when, in a given state, each event leads to at most one successor. A situation is reachable when some sequence of events can actually bring the machine into it. Reachability matters: unreachable states describe behaviour that can never occur, while missing transitions describe events the designer forgot to handle.

A single flat machine, however, becomes unwieldy once a system has many concurrent activities or nested modes of operation. Several extensions were proposed to address this. The most influential is Harel’s Statecharts, which extend finite-state machines with three ideas:

  • Hierarchy. A state may itself contain a complete sub-machine. This allows behaviour to be described at several levels of detail.
  • Orthogonality. A state may contain several regions that are active at the same time. This expresses concurrency directly.
  • Broadcast communication. An event is visible to all regions, so that a single transition may trigger others.

UML State Machines and many industrial modelling tools descend from these ideas.

One observation about Statecharts is essential for what follows. Hierarchy and orthogonality make the description compact. They do not make the underlying state space smaller. Two orthogonal regions with m and n states still denote m × n global configurations. The notation hides the product; it does not remove it.

The distinction between a compact description and a large underlying state space is central to the rest of this tutorial.

The size of the state space, however, is only half of the problem. The other half lies in how the parts of a Statechart interact — and it is to this execution model that we now turn.

4. Interconnection by mutual control

Problem. In classical Statecharts, coordination between the parts of a system is achieved by mutual event exchange. A transition in one region broadcasts an event; the event may trigger transitions in other regions; those transitions may broadcast further events; guards evaluated by one machine may refer to the states of another. Every machine is allowed, in principle, to observe and to command every other. Control is peer-to-peer: there is no distinguished coordinator, and the intended global behaviour is nowhere written down. It emerges from the web of mutual reactions.

A concrete example

Consider a small medical system made of three interconnected devices:

  • an infusion pump;
  • an anti-reflux valve;
  • a blood pressure sensor.

The designer wants to enforce a global behaviour such as:

  • the valve must be opened before the pump starts, and closed after it stops;
  • during pumping, blood pressure must stay below a threshold; if it rises, the pump must be stopped and the valve closed.

Modelled with orthogonal regions and broadcast events, each requirement is implemented by scattering event emissions and guard evaluations across the three machines: the sensor signals the pump, the pump signals the valve, the valve conditions the pump. The result is a web of cluttered causal relationships: the safety policy — a single, coherent intention — is smeared over the internals of every component. No single diagram states it; every diagram implements a fragment of it.

Two structural problems follow.

  • No machine is self-contained. The behaviour of each component depends on events emitted, and states occupied, by the others. A component cannot be understood — nor reused, nor certified — in isolation, because part of its logic lives elsewhere.
  • The global outcome is hard to predict. Since no artefact describes the intended systemic behaviour, the designer can only mentally simulate the mutual reactions — precisely the kind of reasoning that fails as the number of components grows.

When the conversation never ends — or never starts

Mutual control does not merely make models hard to read. It introduces genuine operational risks.

Because a broadcast event may trigger transitions that broadcast further events, a single external stimulus can start a chain reaction of internal reactions. Nothing, in the model itself, guarantees that this cascade converges. Three distinct failure shapes appear:

  • Non-terminating reaction. Two machines may trigger each other in a cycle — A’s transition emits an event that fires B, whose transition emits an event that fires A again. The processing of a single stimulus never completes: the system is trapped inside its own reaction.
  • Livelock. The cascade may terminate step by step, yet the system as a whole keeps oscillating between configurations — pump commanded to start, sensor commanding it to stop, valve reacting to both — perpetually busy, never making useful progress.
  • Deadlock. Symmetrically, each machine may end up waiting for an event that another machine, in its current state, will never emit. The conversation does not loop; it simply stops.

Livelock and deadlock are two faces of the same coin. In a deadlock the system does nothing; in a livelock it does everything except progress. Both are properties of the global behaviour: no inspection of a single machine, however careful, can reveal them, because each machine, taken alone, is perfectly well-formed.

There is a further, subtler difficulty. Exactly how a cascade of broadcast events unfolds — whether reactions occur in the same step or the next, in which order simultaneous transitions fire — is not fixed once and for all: different Statecharts dialects and tools resolve these questions differently. A model may therefore behave differently under different execution semantics, which is a fragile foundation for a safety argument.

None of this means that such systems cannot be built — they are built every day. It means that with peer-to-peer, broadcast-based interconnection, the properties that matter most — will it stop? will it progress? will it stay safe? — are exactly the ones the notation gives no local handle on. We shall give these properties precise names shortly.

5. Composition and the state explosion problem

Real systems are never a single machine. They are many machines evolving together.

Suppose a system contains a motor with three states, a door with two states, and a sensor with four states. The complete system then has

3 × 2 × 4 = 24

possible configurations.

The numbers grow violently. Ten components with only five states each already yield

510 = 9 765 625

configurations.

And this counts only the static configurations. The number of possible executions — the interleavings of component transitions over time — is larger still. Concurrency multiplies not only the states a system can be in, but the paths by which it can get there.

Two distinct costs follow from this growth.

  • Comprehension. No engineer can hold millions of configurations in mind. Intuition, which served well for a single door, fails for the composed system.
  • Verification. Any analysis method that explores the global state space directly inherits this cost.

This exponential growth is known as the state explosion problem. It is not an accident of a particular tool or notation. It is intrinsic to reasoning about a composed system as a single global object.

6. What «correct» means

Before asking whether a system is correct, we must say what correctness is. Requirements on reactive systems fall into two broad families.

Safety. Something bad never happens:

  • the door never closes on a person;
  • the laser never fires while the cover is open;
  • two trains never occupy the same track segment.

Liveness. Something good eventually happens:

  • every request is eventually served;
  • a door that has begun opening eventually becomes open.

Two failure modes deserve to be named explicitly — we have already met both, informally, in the medical example of Section 4:

  • Deadlock — the system reaches a situation from which no further progress is possible. In the mutual-control setting: every machine waits for an event that no other machine will emit.
  • Livelock — the system keeps moving but never makes useful progress. In the mutual-control setting: machines keep reacting to one another’s events in an endless, fruitless exchange.

Note that neither is a property of any single machine. Both arise from the interaction, which is precisely why they escape component-by-component inspection.

Real requirements are usually large conjunctions of many such properties. The important point is structural: a correctness claim quantifies over all possible behaviours, not over a single execution. It is precisely this «for all behaviours» that makes verification hard, and that the state explosion problem attacks.

7. Traditional verification

Formal verification classically follows a fixed workflow:

  1. First, the complete behavioural model of the system is constructed.
  2. Then the desired properties are specified, often in a temporal logic.
  3. Finally, a model checker explores the global state space, searching for a behaviour that violates a property.

When a violation exists, the model checker does not merely answer «no». It returns a counterexample: a concrete scenario that exhibits the fault, step by step. This diagnostic power is one of the great strengths of the approach.

The method has achieved remarkable successes, in hardware design, communication protocols, and safety-critical control. Nor is it defenceless against state explosion: symbolic representations, abstraction, partial-order reduction, and compositional (assume–guarantee) reasoning all extend its reach considerably.

These techniques matter, and some of them will reappear later in a different guise. Yet one structural feature of the workflow remains, independent of any optimisation.

8. The cost of late verification

In the classical workflow, verification is a separate activity, performed after the controller already exists. It answers a yes/no question about a finished artefact.

When the answer is «no», the designer must:

  • interpret the counterexample,
  • locate the underlying flaw in the controller,
  • modify the controller,
  • and repeat the entire verification.

This loop may run many times before the model is finally accepted. Two costs accumulate along the way:

  • Rework. Each failed iteration discards design effort that has to be redone.
  • Late discovery. Faults surface at the end of the process, exactly where they are most expensive to correct.

Beneath both costs lies a single fact. Correctness is something the design is tested for, not something it is built from. And the test operates on the global system, whose size grows exponentially with the number of its parts.

The question, then, is not whether verification works. It is whether correctness must always be established so late, and so globally.

9. Towards another philosophy

Problem. Two questions arise naturally from the discussion so far:

Can a controller be constructed so that its correctness emerges progressively during design, instead of being checked only at the end?

Can a single large global verification be replaced by many small local reasoning steps, justified by the structure of the system itself?

Intuition. Neither question is entirely new. Correct-by-construction design and compositional reasoning have both been studied for decades. Their difficulty has never been the idea; it has been finding a representation in which the idea can actually be applied to reactive controllers.

Such a representation must provide two things:

  • a way to state the intended meaning of each design decision locally, at the level of a single component or subsystem;
  • a guarantee that these local statements compose into global correctness, following the hierarchical structure of the system, without ever building the global product.

Providing exactly this — for reactive controllers, and in a form an engineer can genuinely use — is the motivation behind Part–Whole Statecharts. The next part introduces the model itself.
